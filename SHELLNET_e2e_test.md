# Shellnet end-to-end test

How to reproduce the full Circuit 4 (`WithdrawalInitiated`) proof pipeline against the public Acki Nacki **shellnet** (`https://shellnet.ackinacki.org/graphql`).

This drives both daemons (`bridge-prover-daemon` + `bridge-verifier-daemon`) plus the orchestrator that fires a real withdrawal on shellnet and proves it. Expected wall time: **~10 min** end-to-end on a fresh start.

For the local-devnet variant and the architectural background, see [README.md](./README.md) and [TECHNICAL_README.md](./TECHNICAL_README.md).

---

## 1. Prerequisites

| | Requirement | Notes |
|---|---|---|
| 1 | Rust toolchain | Whatever this repo's `rust-toolchain` pins. |
| 2 | Python 3 | Stdlib only. No `pip install`. |
| 3 | ~16 GB free disk | Circuit 1A/2/4 keys + SRS together. |
| 4 | Network reachable to `https://shellnet.ackinacki.org/graphql` | The orchestrator and both daemons all hit this endpoint. |

No `sold` (Solidity → TVM) compiler needed — all contracts are pre-compiled and bundled under `python/contracts/`.

## 2. One-time setup

All steps run from the repo root.

### 2.1 `tvm-cli`

A pre-built macOS arm64 binary is bundled at `python/bin/tvm-cli`. If it works on your machine, skip ahead. To rebuild it (Linux, other archs, or to upgrade):

```bash
git clone https://github.com/tvmlabs/tvm-sdk.git --branch full_dex_and_bridge_test_with_final_halo2_circuit
cd tvm-sdk
cargo build --release -p tvm_cli
cp target/release/tvm-cli /path/to/this/repo/python/bin/tvm-cli
```

The orchestrator picks up the bundled binary by default; override with `CLI_NAME=/abs/path/to/tvm-cli` if you installed it elsewhere.

### 2.2 Build the three Rust binaries

```bash
cargo build --release \
    -p bridge-event-private-witness-export \
    -p bridge-event-witness-builder \
    -p bridge-event-halo2-prover
```

These produce executables under `target/release/`.

### 2.3 Build the two daemons

```bash
cargo build --release \
    --bin bridge-prover-daemon \
    --bin bridge-verifier-daemon
```

(`scripts/run-bridge-test.sh` also runs this build for you on first launch.)

### 2.4 Generate Circuit 4 keys (~5 min, ~3 GB)

The verifier daemon refuses to start without `params/event_vk.bin`, so generate the Circuit 4 keys **before** launching the daemons:

```bash
cargo run --release --bin bridge-event-halo2-prover -- --selftest
```

This produces `params/event_pk.bin`, `params/event_vk.bin`, and `params/kzg_bn254_20.srs` (the shared trusted-setup SRS).

Circuit 1A and Circuit 2 keys (`primary_*.bin`, `layer_*.bin`) are generated by `bridge-prover-daemon` automatically on its first run.

## 3. Run the test

Open two terminals.

### 3.1 Terminal A — launch the daemons (fresh state)

```bash
BRIDGE_GQL_ENDPOINT="https://shellnet.ackinacki.org/graphql" \
    scripts/run-bridge-test.sh
```

This wipes `proofs/`, `state/`, `logs/` (it keeps `params/`), then launches both daemons in the background and writes PIDs to `logs/test.pids`. Watch their progress:

```bash
tail -f logs/prover_output.log logs/verifier_output.log
```

Wait until you see lines like:

```
bridge_prover_daemon: key block <N>: both proofs generated in ~250s
bridge_verifier_daemon: bootstrapping from seed at ./state/bootstrap_seed.json: seqno=<N>, ...
```

#### How the bootstrap point is chosen

The prover/verifier pair anchor every proof against a **bootstrap seed seq_no** that must satisfy `seqno % (W·P) == 0`, where `W = 128` (`HISTORY_PROOF_WINDOW_SIZE`) and `P = 4` (`THINNING_FACTOR_P`). So bootstrap candidates exist every `W·P = 512` blocks: `…, 1036288, 1036800, 1037312, …`.

In **auto mode** (default, no override) the prover:

1. Reads chain head from GQL at startup.
2. Picks the **next** `512`-boundary strictly past chain head, e.g. head=1036690 → seed=1036800.
3. Sleeps until the chain actually reaches that seed seq_no, then writes `state/bootstrap_seed.json`.
4. The verifier wakes up from polling the seed file and bootstraps from there.

This is why a fresh start takes a few minutes before you see anything: the chain is being given time to catch up to the seed point. On shellnet (~3 blocks/s) the worst case is `512 blocks / 3 b/s ≈ 170s` ≈ 3 min, and on average ~1.5 min.

The orchestrator fires its withdrawal **after** seed is reached, so the event lands a few hundred blocks past the seed — inside the next bundle. The verifier then needs one full bundle prove (~5 min) to advance past the event's anchor before the Circuit 4 step can run.

#### Why fresh-start beats restarting with stale state

If you re-launch the daemons on a `state/` that lags far behind chain head, the verifier has to chain-prove every missed bundle to catch up — that can easily turn a 10-min run into 30+ min. `scripts/run-bridge-test.sh` deliberately wipes `state/` so auto-seed re-anchors near the current chain head every time. Keep state across runs only if you actually want restart-resume behaviour.

#### Forcing a specific seed

Set `BRIDGE_BOOTSTRAP_SEQNO=<N>` on the prover to pin the seed. `<N>` **must** be a multiple of `512`; anything else aborts startup. Useful for reproducing a specific run or for resuming against a known anchor — not needed for normal shellnet runs.

### 3.2 Terminal B — run the orchestrator

```bash
BRIDGE_GQL_ENDPOINT="https://shellnet.ackinacki.org/graphql" \
    python3 python/generate_withdrawals_with_live_event_proving_shellnet.py
```

The orchestrator:

1. Deploys a fresh multisig wallet on shellnet, funded by the `GiverV3` faucet with the canonical amounts (10 NACKL + 100T ECC[2]).
2. Mints ECC[3] (USDC) into the multisig via `USDCBridge.mintAndSend`, signed with the shellnet bridge-owner key bundled at `python/contracts/USDCBridge.shellnet.keys.json`.
3. Waits for a suitable point in the block stream, calls `TokenBridge.initiateWithdrawal(...)`, captures the `WithdrawalInitiated` event.
4. Waits for the verifier to advance to the event's key-block anchor (one bundle past the event ~= 5 min on shellnet).
5. Runs `bridge-event-private-witness-export` → `bridge-event-witness-builder` → `bridge-event-halo2-prover` to produce the Circuit 4 proof.

Intermediate per-run artefacts (multisig keys, partial witnesses) land under `<repo>/work-shellnet/` by default. Override with `WORK_DIR=/abs/path`.

## 4. Success criteria

The orchestrator prints, near the end:

```
[T+10:32] === Step 8: result ===
{
  "verified": true,
  ...
}
```

The proof + result land at:

```
proofs/proof_event_000000.json
proofs/result_event_000000.json
```

Typical timing on a fresh start against shellnet:

| Phase | Wall time |
|---|---|
| Daemon bootstrap + first bundle prove | ~5 min |
| Event capture + verifier catch-up | ~4 min |
| Circuit 4 prove (k=19) | ~3 min |
| **Total** | **~10 min** |

## 5. Stopping cleanly

```bash
scripts/stop-bridge-test.sh
```

Or kill the PIDs in `logs/test.pids` directly. The daemons checkpoint state into `state/` and resume from there if you restart without `run-bridge-test.sh` (which wipes state).

## 6. Configuration reference

| Variable | Default | Purpose |
|---|---|---|
| `BRIDGE_GQL_ENDPOINT` | `http://localhost/graphql` | Shared by both daemons and the orchestrator. Must be set to the shellnet URL for all three. |
| `BRIDGE_BOOTSTRAP_SEQNO` | auto (next `W·P` past head) | Force a specific seed seq_no; must be `% (W*P) == 0`. |
| `WORK_DIR` | `<repo>/work-shellnet` | Per-run intermediates (multisig keys, witnesses). |
| `CLI_NAME` | `python/bin/tvm-cli` | Path to an alternate `tvm-cli`. |
| `RUST_LOG` | `info` | Daemon log level. |

## 7. Troubleshooting

- **`COMPUTE_SKIPPED: empty balance` during multisig deploy** — the faucet amounts in the orchestrator are the canonical ones from `acki-nacki/tests/test_multisig.py`. If you have hacked them smaller, the deployed account has no vmshell to pay compute fees. Restore `value=10_000_000_000_000` + `ecc[2]=100_000_000_000_000`.
- **`Resource not found` when polling the multisig** — the multisig is deployed with its own dapp_id, so the address form is `acc::acc`, not the legacy `0:acc` or the zero-dapp `0000…::acc` form. The orchestrator already uses the self-dapp form; if you are extending it, do the same.
- **`403` from `https://shellnet.ackinacki.org/graphql`** — the reverse proxy rejects the default Python `urllib` User-Agent. The orchestrator overrides it; if you write your own GQL caller, set a custom UA header.
- **Verifier never reaches the event's anchor seq_no** — almost always means the seed was placed far behind chain head and the verifier is chain-proving its way forward bundle-by-bundle. Wipe and restart: `scripts/stop-bridge-test.sh` then `scripts/run-bridge-test.sh` re-seeds at the next `512`-boundary past current chain head, which is usually within a few hundred blocks of where the event will land.
- **`BRIDGE_BOOTSTRAP_SEQNO must be a multiple of 512`** — prover refuses any forced seed that isn't `W·P`-aligned. Drop the override or pick a valid boundary (`echo $((N - N % 512))`).
